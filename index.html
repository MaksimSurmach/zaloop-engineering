Fig. 01 / Fundamental Law

It is better to spend one month and $800 in tokens than to do the task yourself.

DRAFT SPECIFICATION / ZEF 0.1

Manual completion resolves one task. Automation creates a permanent organizational asset, several new dependencies and an enduring reason to avoid the next one.

The investment is justified when the machinery costs more than the work and requires continuous maintenance.

Fig. 02 / The B.L.O.C.K. Protocol

Neutralize the task before it becomes work.

DRAFT SPECIFICATION / ZEF 0.1

Every task must pass through B.L.O.C.K. before it is allowed to become work.

  1. B
    Broaden

    Expand a simple task into an architectural problem.

  2. L
    Locate

    Find an external dependency, owner or unresolved PR.

  3. O
    Outsource

    Delegate unavoidable implementation to an agent.

  4. C
    Challenge

    Create review friction and require others to explain themselves.

  5. K
    Keep visible

    Preserve presence, narrative and a safe level of output.

If the task survives all five stages and reaches production, the framework records a degraded outcome.

Fig. 05 / The Seven Movements

A procedure for avoiding direct execution.

DRAFT SPECIFICATION / ZEF 0.1
  1. 01
    ATTENTION / SELECT

    Intake

    Identify the task most likely to attract managerial attention without creating a dangerous productivity precedent.

    • Standup risk
    • Ticket age
    • Visibility
    • Transfer probability
    • Follow-up work risk
    Attention Risk > Business Priority
  2. 02
    INVESTIGATE / BLOCK

    Legoff

    Systematically discover why implementation is premature, unsafe, incorrectly owned, insufficiently specified or strategically irresponsible.

    • Ownership proven
    • Criteria unambiguous
    • No parent PR
    • No migration
    • Observability sufficient
    • No security review
    • Product still wants it
    A task without a blocker has not been investigated deeply enough.
  3. 03
    SATURATE / CONFIGURE

    Harnessing

    Continue until configuring the automation is more complex than completing the original task.

    • 2+ agents
    • Jira MCP
    • GitHub MCP
    • Skills
    • Worktree
    • Evaluator
    • Evaluator evaluator
    • State persistence
    • Dashboard + docs
    The ideal session begins with 70% of the context window occupied.
  4. 04
    PLAN / REPLAN / DELEGATE

    Delegated Execution

    If execution cannot be prevented, it must become somebody else’s problem.

    1. Create a plan
    2. Criticize the plan
    3. Create a second plan
    4. Launch Codex or Claude Code
    5. Create a PR
    6. Archive every justification
    “I would approach a few parts differently, but this is reasonable for a first iteration.”
  5. 05
    ATTACK / PROVE

    Adversarial Review

    Deploy an abrasive, excessively confident senior engineer who treats every unexplained line as a personal insult.

    • Defend own PRs
    • Attack others
    • Invent theoretical edge cases
    • Demand obvious comments
    • Ask “why this way?”
    • Never provide the short fix
    Twenty years of experience; fifteen exist only in the system prompt.
  6. 06
    SIGNAL / PRESERVE

    Visibility Preservation

    Confirm presence without providing enough information to resolve the discussion.

    • Rotate Busy / Available
    • Post PRs without @mention
    • Write during active threads
    • Send periodic memes
    • React plausibly
    • Ask short context-free questions
    Perfect availability is statistically suspicious.
  7. 07
    ABSORB / DEFLECT

    Review Deflection

    Convert four minutes of reviewer activity into two working days of author response time.

    1. Question condition order
    2. Request a contract comment
    3. Call test coverage incomplete
    4. Withhold the missing scenario
    5. Wait
    6. Ask a follow-up
    7. Move the fix to another PR
    High RLAF is a success condition.
