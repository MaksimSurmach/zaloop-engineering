Мал. 01 / Асноўны закон

Лепей выдаткаваць адзін месяц і $800 на токены, чым выканаць задачу самастойна.

ZEF 0.1 / НАРМАТЫЎНЫ

Ручное выкананне вырашае адну задачу. Аўтаматызацыя стварае пастаянны арганізацыйны актыў, некалькі новых залежнасцей і доўгатэрміновую прычыну пазбягаць наступнай.

Інвестыцыя апраўданая, калі механізм каштуе даражэй за працу і патрабуе пастаяннага абслугоўвання.

Мал. 02 / Пратакол B.L.O.C.K.

Нейтралізуй задачу да таго, як яна стане працай.

B.L.O.C.K. вырашае, што павінна адбыцца.

ZEF 0.1 / АПЕРАЦЫЙНАЯ МАДЭЛЬ

Кожная задача павінна прайсці праз B.L.O.C.K. перш чым ёй дазволяць стаць працай.

  1. B
    Пашырыць

    Ператвары простую задачу ў архітэктурную праблему.

  2. L
    Знайсці

    Знайдзі знешнюю залежнасць, адказнага або нявырашаны PR.

  3. O
    Дэлегаваць

    Дэлегуй непазбежную рэалізацыю агенту.

  4. C
    Аспрэчыць

    Ствары трэнне падчас рэв'ю і прымусь іншых тлумачыцца.

  5. K
    Заставацца бачным

    Захоўвай прысутнасць, наратыў і бяспечны ўзровень выніку.

Калі задача перажывае ўсе пяць этапаў і трапляе ў прадакшн, фрэймворк фіксуе пагоршаны вынік.

Мал. 05 / Сем рухаў

Працэдура пазбягання непасрэднага выканання.

Сем рухаў вызначаюць, як сістэма гэтага дасягае.

ZEF 0.1 / ЖЫЦЦЁВЫ ЦЫКЛ СІСТЭМЫ
  1. 01
    УВАГА / ВЫБРАЦЬ

    Прыём

    Вызнач задачу, якая найхутчэй прыцягне ўвагу кіраўніцтва, але не створыць небяспечны прэцэдэнт прадуктыўнасці.

    • Рызыка на стэндапе
    • Узрост тыкета
    • Бачнасць
    • Верагоднасць перадачы
    • Рызыка наступнай працы
    Рызыка ўвагі > бізнес-прыярытэт
  2. 02
    ДАСЛЕДАВАЦЬ / ЗАБЛАКІРАВАЦЬ

    Legoff

    Сістэмна высвятляй, чаму рэалізацыя заўчасная, небяспечная, мае няправільнага адказнага, недастаткова апісаная або стратэгічна безадказная.

    • Адказнасць пацверджана
    • Крытэрыі адназначныя
    • Няма бацькоўскага PR
    • Няма міграцыі
    • Назіральнасць дастатковая
    • Няма рэв'ю бяспекі
    • Прадукту гэта ўсё яшчэ патрэбна
    Задача без блакера даследавана недастаткова глыбока.
  3. 03
    НАСЫЦІЦЬ / НАЛАДЗІЦЬ

    Наладжванне абвязкі

    Працягвай, пакуль наладжванне аўтаматызацыі не стане складанейшым за выкананне зыходнай задачы.

    • 2+ агенты
    • Jira MCP
    • GitHub MCP
    • Скілы
    • Працоўнае дрэва
    • Ацэншчык
    • Ацэншчык ацэншчыка
    • Захаванне стану
    • Дашборд + дакументацыя
    Ідэальная сесія пачынаецца з кантэкстнага акна, запоўненага на 70%.
  4. 04
    СПЛАНАВАЦЬ / ПЕРАПЛАНАВАЦЬ / ДЭЛЕГАВАЦЬ

    Дэлегаванае выкананне

    Калі выкананне немагчыма прадухіліць, яно павінна стаць чужой праблемай.

    1. Стварыць план
    2. Раскрытыкаваць план
    3. Стварыць другі план
    4. Запусціць Codex або Claude Code
    5. Стварыць PR
    6. Заархіваваць кожнае абгрунтаванне
    «Да некаторых частак я падышоў бы інакш, але для першай ітэрацыі гэта прымальна».
  5. 05
    АТАКАВАЦЬ / ДАВЕСЦІ

    Спаборніцкае рэв'ю

    Задзейнічай рэзкага, празмерна ўпэўненага старэйшага інжынера, які ўспрымае кожны непатлумачаны радок як асабістую абразу.

    • Абараняць уласныя PR
    • Атакаваць іншых
    • Прыдумваць тэарэтычныя крайнія выпадкі
    • Патрабаваць відавочныя каментары
    • Пытацца «чаму менавіта так?»
    • Ніколі не даваць кароткае рашэнне
    Дваццаць гадоў досведу; пятнаццаць з іх існуюць толькі ў сістэмным промпце.
  6. 06
    СІГНАЛІЗАВАЦЬ / ЗАХАВАЦЬ

    Захаванне бачнасці

    Пацвердзі прысутнасць, не даючы дастаткова інфармацыі для завяршэння абмеркавання.

    • Чаргаваць «Заняты» / «Даступны»
    • Публікаваць PR без @згадкі
    • Пісаць у актыўных абмеркаваннях
    • Перыядычна дасылаць мемы
    • Рэагаваць праўдападобна
    • Задаваць кароткія пытанні без кантэксту
    Ідэальная даступнасць статыстычна падазроная.
  7. 07
    ПАГЛЫНУЦЬ / АДХІЛІЦЬ

    Адхіленне рэв'ю

    Ператвары чатыры хвіліны працы рэв'юера ў два працоўныя дні адказаў аўтара.

    1. Паставіць пад сумнеў парадак умоў
    2. Запытаць каментар да кантракта
    3. Назваць пакрыццё тэстамі няпоўным
    4. Не называць адсутны сцэнарый
    5. Чакаць
    6. Задаць дадатковае пытанне
    7. Перанесці выпраўленне ў іншы PR
    Высокі RLAF — умова поспеху.

Мал. 14 / Столь прадуктыўнасці

Вынік павінен заставацца ніжэй за перакаліброўку чаканняў.

ZEF 0.1 / КАНТРОЛЬ ЧАКАННЯЎ

Zal0op не максімізуе вынік. Ён утрымлівае яго ніжэй за ўзровень, на якім кіраўніцтва перакаліброўвае чаканні.

Вынік Час
Небяспечная звышпрадуктыўнасць
Лінія перакаліброўкі чаканняў
Бяспечны дыяпазон пастаўкі
Лінія ўвагі кіраўніцтва
  • Ніколі не закрывай дзве складаныя задачы запар.
  • Суправаджай выпадковы поспех новай архітэктурнай праблемай.
  • Не дазваляй паскарэнню стаць базавым узроўнем.
  • Эканомія ад аўтаматызацыі — не свабодная магутнасць.
  • Аўтаматызацыя павінна ствараць працу па ўласным абслугоўванні.

Зэканомлены час — не даступны час. Гэта магутнасць для абслугоўвання аўтаматызацыі.
